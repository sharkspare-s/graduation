# EV-UAV 事件追踪扩展方案

这个仓库原始能力是“事件级前景分割”，不是完整追踪器。要把它扩成事件相机追踪系统，推荐采用外接式改造，而不是改 `dataset/`、`model/`、`utils/` 里的原始实现。

## 改造原则

1. 保留原始模型：
   `model/evspsegnet.py` 继续作为事件前景提取网络，不改网络结构和训练代码。
2. 保留原始数据管线：
   `dataset/basedataset.py` 的体素化和 `custom_collate` 继续复用，不重写核心稀疏卷积输入流程。
3. 只在外层新增追踪逻辑：
   新增事件流适配、检测后处理、轨迹关联、可视化脚本。

## 当前仓库能直接复用的部分

- `run_realtime.py`
  已经说明作者支持从事件流做实时推理。
- `dataset/basedataset.py`
  已经实现事件点云到稀疏体素的转换。
- `model/evspsegnet.py`
  输出每个体素/事件的前景概率。

这意味着你不需要先把项目改成“检测网络”，可以先把它当作“事件前景分割 + 后处理追踪”的系统。

## 新增文件

- `run_tracking.py`
  独立追踪入口，不修改原始实时脚本。
- `tracking/postprocess.py`
  把分割分数图转换成连通域检测结果。
- `tracking/tracker.py`
  做简单的多目标轨迹管理。

## 系统流程

1. 事件相机输出一个时间窗内的事件。
2. 复用原始体素化流程，将事件变成 `SparseConvTensor`。
3. 用 `EV-SpSegNet` 得到每个事件的前景概率。
4. 将高分事件投影到二维像素平面。
5. 用连通域把前景事件聚成一个或多个检测框。
6. 用最近邻关联维护轨迹 ID。
7. 输出带 ID 的目标框，形成追踪结果。

## 这和“事件检测”/“事件追踪”的关系

- 这个仓库原始任务更接近事件小目标分割。
- 现在新增的方案，本质上是：
  `事件分割器 -> 目标生成器 -> 目标追踪器`
- 这是最稳妥的第一步，因为不需要重训整个模型头。

## 如果你想进一步做成更标准的事件追踪系统

可以继续沿这条外接路线迭代，而不是立刻改底层源码：

1. 单目标追踪：
   在第一帧人工初始化或用检测自动初始化，只跟踪一个 UAV。
2. 多目标追踪：
   保留当前多检测框输出，换更强的关联器。
3. 时序增强：
   在 `tracking/` 里加入短时轨迹缓存、速度门控、卡尔曼滤波。
4. 目标状态输出：
   输出 `track_id / center / bbox / confidence / velocity`。
5. 轨迹级滤波：
   用连续命中数、消失容忍窗口抑制闪烁误检。

## 什么时候才需要改原始源码

只有在下面几种情况下才建议动原始网络：

1. 你要把“分割头”改成“检测头”。
2. 你要做联合训练，让模型直接学 `bbox + id embedding`。
3. 你要把时间窗建模从后处理迁移到网络内部。

在毕业设计阶段，先做外接式追踪比直接改主干更稳，也更容易写清楚系统演进路线。

## 运行示例

```bash
python run_tracking.py --config configs/evisseg_evuav.yaml --checkpoint path/to/best.pt
```

如果你的 `yaml` 里还是作者机器上的绝对路径，直接用 `--checkpoint` 覆盖即可。
